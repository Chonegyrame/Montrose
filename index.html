01Short Version

This document describes an AI-driven marketing stack built around a centralized context repository that every agent reads from and every validated learning writes back to.

The stack consists of specialized agents such as: Paid Ads, Organic Social, Email, and SEO/Content (more can of course be added). Each agent is triggered by a campaign brief and each constrained by the same compliance rules, brand voice, and ICP data. No agent publishes anything without human review. No insight updates the repository without human validation.

The result is a system that does not just produce content faster. It gets more accurate over time. Each campaign cycle sharpens the repository. Each sharpened repository improves the next generation. For a regulated financial services company like Montrose, this means higher output volume, consistent brand voice, and compliance managed at the architecture level.

02The Problem: Disconnected Tools Don't Build a System

Most marketing teams that adopt AI end up with the same structural problem. A copywriter uses ChatGPT with a loose, undocumented prompt. A social manager runs a different tool with no prompt structure at all. An ads specialist maintains personal templates in a private document. The result is higher output volume without higher consistency, and without consistency there is no feedback loop worth measuring. The brand says slightly different things across channels, and because no shared system tracks what works, the same mistakes repeat. Approved claims get paraphrased in ways nobody has reviewed. In financial services, where every public claim carries regulatory risk, that is a compliance problem.

The solution is a shared knowledge layer: a full stack marketing system that uses a context repository that every tool reads from, writes back to, and gets evaluated against.

03The Solution: A Context Repository

The context repository is the single most important architectural decision in this stack. It is not a brand guidelines PDF or a shared folder. It is a structured, versioned knowledge base that functions as the collective memory for every agent in the system.

How it works in practice: When an agent is triggered (say, to produce a LinkedIn post), it doesn't start from a blank prompt. It first reads the relevant sections of the repository: who is the target audience, what tone does the brand use, what has performed well in this channel before, what claims are approved. It generates output based on that context. After the content runs and performance data comes in, validated learnings are written back into the repository, so the next time that agent runs, it starts from a stronger foundation.

This is how the system compounds. Each campaign cycle, the repository gets sharper. What worked gets encoded. What failed gets flagged. Over time, it becomes a precise operational record of what the brand actually is and what actually drives results.

  • Brand voice -tone, vocabulary, forbidden phrases, and annotated examples of on-brand vs. off-brand writing
  • ICPs & personas (Ideal Customer Profiles) -who the target customers are, what problems they have, and what language resonates with them
  • Product USPs & approved claims (Unique Selling Points) -what Montrose can credibly say, with evidence, and what it cannot
  • Compliance rules -regulatory constraints and required disclaimers, always current
  • Offer positioning -active campaigns, current CTAs (calls to action), seasonal context
  • Channel playbooks -how each platform works, format specs, cadence rules
  • Prompt modules -reusable, versioned prompt components that agents pull from per task type
  • Campaign learnings & performance insights -a growing record of what has worked, by segment, channel, and message type

04The Architecture: Three Layers

Layer 1: Context Repository

The persistent memory layer. Every section is versioned, has a clear owner, and a defined review cadence. Content enters through a validation gate to ensure only accurate, validated learnings update the collective memory, though not automatically. Content enters only after human review. This is the only layer that no agent can write to on its own.

Layer 2: Agent Layer

Four specialized agents handle content production. Each one has a defined scope, reads the relevant sections from the repository before generating anything, and routes all output through a human QA (quality assurance) checkpoint before content leaves the system.

Agents do not communicate with each other directly. This is a deliberate design decision. All inter-agent communication happens through the repository in a hub-and-spoke model. If agents were to communicate peer-to-peer, you'd get compounding effects that are difficult to track, measure and debug. Instead: if the Paid Ads Agent surfaces a valuable insight, it gets validated, written to the repository, and read by the Email Agent on its next run. The loop takes slightly longer, but every output remains fully auditable and the system stays controllable.

Each agent is powered by an AI API (such as Claude or GPT). The operator triggers a specific agent by submitting a brief: a short, structured prompt that defines the task. For an email campaign, a brief might read: "Write a nurture email for ICP_01 (first-time investor), objective: drive to fund landing page, tone: educational." The agent combines that brief with everything it reads from the repository (ICP profiles, approved claims, active offer, previous campaign learnings) and generates a complete draft. The brief determines which agent runs and what the output targets; the repository determines the quality and consistency of what gets produced.

Paid Ads Agent

Meta · Google Ads · LinkedIn
▶

Takes a campaign brief and writes the actual ad copy. Before generating anything, it scrapes the repository for:

  • What the target ICP cares about and responds to
  • What offer is currently live and how it's positioned
  • What claims are approved -and which phrases are forbidden

Output: 3–5 headline variants, 2–3 body text versions, and 2 CTA options, within platform character limits, compliance-flagged, and tagged by segment and format. Lands in QA ready for a human to review and push live.

Organic Social Agent

LinkedIn · Instagram
▶

Writes platform-native posts: a LinkedIn thought leadership piece, an Instagram caption, a short-form hook. Before writing, it scrapes the repository for:

  • What angles have already been published, to avoid repetition
  • What has performed well with the target segment
  • Channel playbook rules for tone, format, and cadence

Output: post copy adapted per platform, suggested hashtags, and a recommended publish window.

Email Agent

Nurture · Activation · Re-engagement
▶

Writes full email sequences (subject line, preview text, and body) for a defined segment and objective. Before writing, it scrapes the repository for:

  • Persona data -pain points, language patterns, decision triggers per ICP
  • Active offer and how it maps to that segment's priorities
  • Previous campaign learnings for this sequence type

A nurture sequence for a first-time investor looks different from a re-engagement email to a dormant lead, even if the underlying offer is the same. Output includes 2–3 subject line variants and segment tags that connect back to CRM (Customer Relationship Management) targeting.

SEO / Content Agent

Blog · Landing Pages · Articles
▶

Writes longer-form content: a blog post on pension savings, a landing page for a specific fund, a guide for first-time investors. Before writing, it scrapes the repository for:

  • ICP pain points and the questions they're actually searching for
  • Product USPs and approved claims relevant to the topic
  • Brand voice guidelines to ensure consistency across the stack

This agent requires more editorial input than the others. Long-form content needs a clearer brief and closer human review before publishing.

Token Economy

Agents do not read the full repository on every run. Each agent pulls only the sections relevant to its task: the Paid Ads Agent does not need the SEO prompt modules, and the Email Agent does not need the channel playbook for Instagram. Repository sections are tagged by agent relevance, and overall this keeps per-generation token costs predictable and avoids hitting context window limits as the repository grows over time.

Prompt modules are versioned partly for quality reasons and partly for cost reasons, a tighter, well-structured module generates better output in fewer tokens than a loose prompt. As the system matures, prompt efficiency could become a metric alongside other measurements such as QA pass rate and content velocity.

Layer 3: Distribution & Feedback

The execution and learning layer. Approved content moves from QA to scheduling and publishing. Performance data flows in from all active platforms. The feedback loop closes through a governed validation process: insights that clear a threshold are written back to the repository; insights that don't are logged for later review. A/B testing can be layered on top of this foundation to further optimize performance, systematically testing hypotheses across copy, format, and segment before encoding winners into the repository.

Two governance checkpoints are non-negotiable: the human QA gate before content publishes, and the human validation gate before insights update the repository.

06How the Feedback Loop Improves the System

This is where most AI marketing implementations stop short. They generate content well but don't build institutional memory.

In this system: aggregate performance data is ingested from ad platforms, email tools, and web analytics. No personal data enters the loop, only campaign-level metrics. Examples: cost-per-lead by ICP segment from Meta Ads, click-through rate by content angle from LinkedIn, open rate and CTR by subject line variant from email sequences. Patterns that clear validation thresholds are extracted as structured learnings and written back to the repository. Prompt modules that underperform are flagged and revised. Campaign learnings accumulate as a queryable, operational asset.

QA Pass Rate weeks 1–8 · target >80%
Each cycle, the system is sharper than the last. What works can be implemented into the repository to increase efficiency and accuracy. What fails gets flagged and reviewed. Every agent benefits from the full performance history of every other agent.

07Why This Design Fits Montrose

  • Coordinated output across channels - Four agents can run simultaneously, each reading from the same strategic context, producing consistent and tailored marketing.
  • Compliance is structural, not manual - Rules live in the repository and are read before every generation. Human QA reviews against those same rules before publishing.
  • Built to improve - Every validated learning gets written back to the repository, so each campaign cycle starts from a stronger foundation than the last.
  • Audit-ready by design - Version control and clear ownership on every repository section makes governance explicit and easy to report on within the DNB Carnegie group structure.

08Final Takeaway

This is how I would set up an AI marketing stack that continuously improves and allows Montrose to publish tailored content across channels in an efficient manner. I would like to be the person who builds it.
